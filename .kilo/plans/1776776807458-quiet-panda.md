# Plan: Fix localStorage Stale Data Issue

## Problem Diagnosis

**Symptom:** After installing the app (Add to Home Screen), the UI shows old localStorage data instead of fresh data from Supabase, even though sync is enabled.

**Root Cause:** The `fetchAllData()` method in `supabaseSync.js` writes fresh data to localStorage but **does not dispatch the `supabase-data-changed` event**. The AppContext only updates its React state when it receives that event. So:

1. App loads → state initialized from stale localStorage
2. `fetchAllData()` runs → overwrites localStorage with fresh data
3. No event dispatched → React state never updates
4. UI remains stuck on old data

**Why it's worse on mobile/PWA:**
- No service worker (app is not a PWA yet)
- "Add to Home Screen" just creates a bookmark; localStorage persists
- First load after install reads whatever was in localStorage from previous visits
- No visual indication that sync is in progress

---

## Current Architecture Review

### Data Flow (As-Designed)
```
localStorage (persistent) ← writeCache() → Supabase (server)
     ↑                                     ↓
React state init                         fetchAllData()
     ↓                                     ↓
   UI          ← dispatchEvent('supabase-data-changed') ← handleRealtimeChange()
```

**Gap:** `fetchAllData()` writes to localStorage but does not trigger the event, so React state never refreshes.

### Caching Strategy (Offline-First)
- **localStorage**: Primary data source for instant reads (offline-capable)
- **Supabase**: Sync target for multi-device persistence
- **Cache version `'v2'`**: Clears all localStorage on version mismatch (line 166-179)
- **No service worker**: App is not a PWA; "Add to Home Screen" doesn't change behavior
- **Realtime subscriptions**: Push updates when online (but unreliable on some networks)

## Recommended Fixes (Priority Order)

### Fix 1: Connect fetchAllData() to React state updates (Critical)
**File:** `src/lib/supabaseSync.js`
**Location:** `fetchAllData()` method, after `writeCache()` call
**Change:** Dispatch `supabase-data-changed` event for each table after fetching so AppContext refreshes state from the updated localStorage.

```javascript
async fetchAllData() {
  const tables = [...];
  for (const table of tables) {
    try {
      const { data } = await supabase.from(table).select('*');
      const cleanData = (data || []).map(item => mapToCamelCase(item));
      writeCache(table, cleanData);
      localStorage.setItem('kk_last_refresh', String(Date.now()));
      console.log(`📥 Fetched ${table}: ${cleanData.length} records`);

      // NEW: Notify AppContext to refresh state from localStorage
      window.dispatchEvent(new CustomEvent('supabase-data-changed', {
        detail: { table, event: 'UPDATE', record: null }
      }));
    } catch (error) {
      console.error(`❌ Failed to fetch ${table}:`, error.message);
    }
  }
}
```

**Impact:** UI will immediately reflect server data after `init()` completes.

## Should We Remove localStorage?

**Short answer:** No, keep it. localStorage is essential for offline-first architecture.

**Why keep localStorage:**
1. **Offline functionality** — app works without network
2. **Instant reads** — no loading spinner on every page
3. **Queue operations** — changes sync when back online
4. **User experience** — perceived performance

**What needs fixing:**
- The **sync direction** must be: Supabase → localStorage → React state, but state should **refresh after fetch completes** (currently broken).
- Add **visual feedback** ("Updating from cloud...") so user knows data is refreshing.

## Risks & Considerations

- **Race condition:** Multiple simultaneous realtime events vs fetchAllData — already handled by `isLocalChange` flag
- **Event storm:** Dispatching 9 events from `fetchAllData()` might cause 9 re-renders; can batch with `setTimeout(..., 0)` or combine into single event
- **Offline:** `fetchAllData()` fails silently (logged), app continues with localStorage data — this is acceptable
- **LocalStorage limits:** ~5MB; if exceeded, sync breaks (monitor with try/catch already in place)
